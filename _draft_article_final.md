如果你没用过 AI 编程工具，可能不知道我说的 Skill 是什么。

先解释一下。Skill 是给 AI 编程助手（比如 Cursor、Claude Code 这些）装的一套说明书。你装了某个 Skill 之后，当你跟 AI 说"帮我生成一个 PPT"的时候，AI 就知道该用哪个库、封面怎么排、配色怎么搭、输出什么格式。不是让 AI 瞎猜，是给它一套标准操作流程。

不装 Skill 的 AI 长这样。你说"做个 Excel"，它拿起 openpyxl 就写，排完透视表丢了也不管。装了 Skill 之后，它会说"我知道怎么处理这个"，然后走一份正经的工程方案。

好，Skill 是什么讲完了。来说今天的主角。

## 01 为什么文档生成一直做不好

说来说去，写代码好说。文本对了就能跑。画图也好说，像素输出了就能看。但文档不行。

一个 Excel 文件，里面藏着一堆东西。公式、条件格式、数据透视表、VBA 宏。拿 openpyxl 去改，稍微复杂的格式直接静默丢掉了。一个 PPT，拿 python-pptx 去排，排出来你自己都不想看。一个 Word，拿 python-docx 去生成嵌套表格，样式直接炸掉。

这几个轻量级 Python 库，设计之初是用来给人类开发者手写脚本的。对格式容错、嵌套结构这种事，它们不在乎。

MiniMax 做了什么事呢。他们把 PPT、PDF、Excel、Word 四个文档 Skill 开源了。核心思路是不靠那些半吊子的 Python 库，直接在底层动手。

同一个问题，不同文档给了不同的解法。但思路一致：找最底层、最官方的工具链，不要中间层套壳。

## 02 技术路线

Excel 放弃了 openpyxl，直接在 XML 层面操作。xlsx 文件本质是个 zip 包，解压出来一堆 XML。直接改 XML 节点，改完重新打包。文件里的数据透视表、VBA 宏、跨表公式，一个都不会丢。生成完还会自己跑一遍公式检查，专门查 #REF! #VALUE! 这种低级错误。

Word 放弃了 python-docx，改用微软官方的 .NET OpenXML SDK。嵌套表格、页眉页脚、修订追踪，全都能保留。

PPT 有 18 套配色 4 种风格。封面、目录、内容页、分隔页、总结页，结构是完整的。不是那种生成出来比你自己做的还丑的水平。

PDF 封面用 HTML + CSS 渲染保设计质量，正文用 ReportLab 处理保稳定性。15 种封面样式，字体颜色自动适配，直接打印没问题。

还做了一个 Execute → Evaluate → Fix 的闭环。生成完了自己检查结构、公式、格式，发现问题自己存成修复案例。不是静止的代码，自动跑了自动修，质量一次比一次稳。

这个机制是这套 Skills 跟市面上其他文档生成方案最大的区别。大部分方案是「生成完就结束」，它多跑了一步检查和修复。

## 03 说实话的短板

这东西目前适合三类人：做 freelancer 的（方案、报价单、合同）、做课程的（课件、讲义）、想做内容分发的（文章转 PDF 资源包）。

但如果你日常不跟 PPT、Excel、Word 打交道，那这套 Skills 技术上很好，但实操上近期用不上。我自己的情况就是日常写文章配图发公众号，不存在「生成 PPT 发给客户」的场景。

另外部署需要 PptxGenJS、.NET SDK、pandas 这些工具链。每个 Skill 的依赖不一样，PPT 和 PDF 的依赖相对轻，Excel 和 Word 的重一些。不是一条命令装完就完事的项目。

目前还在 Beta，GitHub 上 56 个 Open Issue，API 和配置格式可能会有变化。生产环境用的话需要关注版本变更。
